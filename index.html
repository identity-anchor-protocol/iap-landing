Why this exists

Agents act more and more autonomously. Trust requires continuity.

Autonomous agents increasingly make decisions, manage memory, hold credentials, and interact with humans and other systems over time.

Yet most agents today have no verifiable continuity. If their environment changes, if their prompts mutate, or if their key custody shifts, there is no cryptographic way to prove that the current instance is the same identity that acted before.

IAP addresses a narrow but foundational problem: binding an agent identity to a key, and binding evolving local state to that identity in an append-only, verifiable way.

This is not about making models deterministic. It is about making identity and state continuity verifiable.

Public beta · Current scope

What the beta actually does

The current beta is intentionally minimal. It focuses on one thing: binding an agent identity to a cryptographic key and certifying append-only state continuity through the registry.

Users operate locally through iap-agent. The registry acts as the online issuance authority. Certificates can be verified offline using the published registry public key.

The Atlas session demonstrated this flow end-to-end. It was the first documented case of an autonomous agent executing it independently. Read the full transcript.

Included in beta

  • Project-local or global agent identity (Ed25519-based)
  • Append-only local state tracking via AMCS
  • Identity-anchor and continuity certificate issuance
  • Offline verification using the registry public key
  • Quota-gated access through accounts and API keys

Not claimed in beta

  • Deterministic LLM outputs
  • Behavioral continuity proofs
  • Semantic truth guarantees
  • Public self-service signup

What the current beta actually checks

The registry does not inspect your raw files. It evaluates signed requests and certificate continuity for the selected agent_id.

Identity anchor

  • Binds an agent_id to the agent public key
  • Issued once for a new agent identity
  • Required for strict continuity verification

Continuity certificate

  • Certifies a later memory_root for the same agent_id
  • Requires a strictly increasing sequence
  • Can be verified offline with the registry public key

If a continuity request fails because the submitted sequence is stale, use iap-agent registry status --json to inspect the latest registry-side sequence for that agent_id.

Version note: the current beta deployment keeps the protocol at IAP-0.1. The continuity certificate subtype remains IAP-Continuity-0.2 for compatibility with the existing issued chain and verification vectors. That subtype is a compatibility label, not a separate protocol generation.

Protocol specification

IAP is a minimal protocol for tamper-evident agent state continuity and offline verification. It binds an agent identity to an Ed25519 keypair, produces canonicalized commitments, and supports independent verification.

The beta protocol line is IAP-0.1. If you see IAP-Continuity-0.2 inside a continuity certificate, that is an intentional compatibility subtype retained for the deployed v0.1.x ecosystem.

What IAP provides

  • Cryptographic identity binding (Ed25519)
  • Canonicalized state commitments (stable hashes)
  • Append-only continuity tracking
  • Offline verification using the Registry public key

What IAP does not provide

  • Deterministic LLM outputs
  • Behavioral continuity proofs
  • Public self-service signup

Quickstart

Use the current beta flow: install the CLI, create a project-local identity, append state, issue certificates, and verify offline.

python -m pip install -U iap-agent
iap-agent init --project-local --show-public --json
iap-agent setup --registry-base "https://registry.ia-protocol.com" --registry-api-key "iapk_live_optional" --account-token "iapt_live_optional" --check --json
iap-agent amcs append --amcs-db ./amcs.db --file ./AGENT.md --file ./SOUL.md --json
iap-agent anchor issue --registry-base "https://registry.ia-protocol.com" --agent-name "Atlas" --payment-provider auto --open-browser --wait --json
iap-agent continuity request --registry-base "https://registry.ia-protocol.com" --amcs-db ./amcs.db --json
iap-agent continuity wait --registry-base "https://registry.ia-protocol.com" --request-id <request_id> --timeout-seconds 600 --poll-seconds 5 --json
iap-agent continuity cert --registry-base "https://registry.ia-protocol.com" --request-id <request_id> --output-file ./continuity_record.json --json
iap-agent verify ./continuity_record.json --registry-base "https://registry.ia-protocol.com" --json

For a brand-new agent, also save the identity-anchor certificate. If the continuity request returns PENDING, run iap-agent continuity pay ... before waiting.

Autonomous agent integration

IAP can be executed from within an agent session

The full IAP flow — init, append, anchor, continuity, verify — runs entirely through the CLI. An agent with shell access can execute the issuance and verification steps directly, once credentials have been provisioned. The Atlas session is a documented example of this executed end-to-end.

The pattern: the agent generates and holds its own Ed25519 keypair locally, appends its defining files to AMCS, and requests certificates from the registry. The private key never leaves the agent's local environment. Verification is offline and does not require contacting the registry at runtime.

What to track

AMCS tracks any files that constitute the agent's identity and state — typically:

  • Identity and persona definition files
  • Persistent memory files
  • Tool configuration or instruction files

The registry receives only the memory_root hash — not the file contents. Raw state stays local.

Continuity on a schedule

Once anchored, an agent can request continuity certificates periodically as its tracked state evolves. Each certificate certifies a memory_root at a specific sequence, building an append-only chain verifiable offline.

A continuity certificate is the cryptographic answer to: "Is the current instance the same identity that acted at sequence N?"

Key custody

The Ed25519 private key is generated locally and never transmitted to the registry. If the key is lost, identity continuity cannot be re-established from the same agent_id. Back up ~/.iap_agent/identity/ (global) or .iap/identity/ (project-local, when initialized with --project-local).

The commands are identical to the quickstart above. The Atlas session — the first documented case of autonomous IAP self-registration — is available in full: read the transcript.

Trust & Security

  • Offline verification: verify signatures without contacting the Registry.
  • Minimal surface: identity, continuity, lineage—no governance or behavioral scoring.
  • Explicit boundaries: IAP attests commitments; it does not guarantee semantic truth or runtime integrity.
  • Deterministic encoding: canonical JSON + Ed25519 signing for stable verification.
